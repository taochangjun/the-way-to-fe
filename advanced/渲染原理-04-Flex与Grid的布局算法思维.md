# 渲染原理 04：Flex 与 Grid 的布局算法思维

> 目标：不只会写属性，而是知道浏览器在“怎么算”。

---

## 1. Flex 的本质：剩余空间分配

Flex 可以理解为一维空间分配器。

核心过程（简化）：

1. 确定主轴方向（`flex-direction`）
2. 计算每个子项的基础大小（`flex-basis`/宽高）
3. 算剩余空间（容器尺寸 - 子项总基础尺寸）
4. 按 `flex-grow` 分配多余空间
5. 空间不够时按 `flex-shrink` 压缩

---

## 2. 为什么 `flex: 1` 能等分

`flex: 1` 常等价于：

```css
flex-grow: 1;
flex-shrink: 1;
flex-basis: 0%;
```

重点是 `flex-basis: 0%`：  
先把子项基础占位当成 0，再按 grow 比例分配，所以常呈现“平均分”。

### 2.1 大白话类比：先清空座位，再按人数分票

把容器想象成一排座位：

- `flex-basis` = 每个人“预占”几个座位
- `flex-grow` = 剩余座位怎么按比例分
- `flex-shrink` = 座位不够时谁压缩更多

`flex: 1` 常见效果像是：  
“大家先不预占（basis=0），剩余座位按同样权重平分”。

---

## 3. Grid 的本质：二维轨道求解

Grid 不是“更强的 Flex”，而是另一类问题求解器：

- Flex 关注一维分配
- Grid 关注二维切分（行 + 列）

当你写：

```css
grid-template-columns: 200px 1fr 200px;
```

浏览器在做的是轨道宽度约束求解：固定值先占，剩余给 `fr`。

---

## 4. `fr`、`minmax()`、`auto-fit` 的关系

常见组合：

```css
grid-template-columns: repeat(auto-fit, minmax(220px, 1fr));
```

大白话：

- `220px`：每个卡片的最低体面宽度
- `1fr`：空间够时可以拉伸
- `auto-fit`：一行放得下几个就自动放几个

这就是响应式网格的高效写法。

### 4.1 为什么这套写法对新手友好

因为你不需要手动写很多断点去规定“现在 2 列、现在 3 列”。  
浏览器会根据 `minmax` 的下限自动决定一行放几个卡片。

你只需定义：
- “卡片最低不能小于多少”（可读性底线）
- “有多余空间就拉伸”（视觉填充）

---

## 5. Flex vs Grid 的实战分工（再强化）

- 导航栏、按钮组、输入区对齐 -> Flex
- 页面骨架、卡片矩阵、仪表盘区块 -> Grid

最稳组合：

- Grid 搭骨架
- Flex 做组件内部排布

---

## 6. 算法视角下的常见误区

1. 拿 Flex 硬做复杂二维布局  
   -> 容易嵌套过深、难维护

2. 拿 Grid 做简单一维对齐  
   -> 代码偏重，不够直观

3. 不理解 `basis` 就盲用 `flex: 1`  
   -> 容易出现“为什么不是我想象的比例”

4. 忘记给 flex 子项设置 `min-width: 0`  
   -> 长单词/长链接把布局撑爆

5. 把 `gap` 和 `margin` 混着乱用  
   -> 间距来源混乱，排查困难

---

## 7. 一句话总结

写 Flex/Grid 的正确姿势是：  
先想“这是一维问题还是二维问题”，再写属性。  
这样你是在“选算法”，不是在“碰运气”。

---

## 8. Flex 进阶：`flex-basis`、`width`、`min-width` 的关系

新手常见困惑：写了 `flex: 1`，为什么某个子项还是不缩？

常见原因：
- 子项内容太长（默认 `min-width: auto`）
- `flex-basis` 与 `width` 同时存在时优先级理解错误

实战建议：
- 在需要可压缩的场景补 `min-width: 0`
- 明确 `flex-basis` 才是 flex 算法的基础尺寸输入

---

## 9. Flex 分配的两个阶段（简化）

可以把 Flex 分配看成两轮：

1. **先看是否有剩余空间**  
   有 -> 按 grow 分

2. **空间不足时是否允许压缩**  
   不够 -> 按 shrink 压

所以不是“只看 `flex: 1` 就完了”，还要看容器大小、内容最小尺寸约束。

---

## 10. Grid 进阶：显式网格与隐式网格

- **显式网格**：你主动声明的轨道（`grid-template-columns/rows`）
- **隐式网格**：内容超出显式定义后，浏览器自动补出来的轨道

这解释了为什么有时你只写了两列，却出现第三列/新行轨道。

可用 `grid-auto-rows` / `grid-auto-columns` 控制隐式轨道尺寸。

---

## 11. `minmax()` 的工程价值

`minmax(220px, 1fr)` 本质是在做“下限保护 + 上限弹性”。

它比固定列宽更稳，因为：
- 小屏不会挤得不可读（有最小宽度）
- 大屏不会留太多空白（可弹性拉伸）

一个常用模板：

```css
grid-template-columns: repeat(auto-fit, minmax(220px, 1fr));
```

你可以把 `220px` 当成“设计稿里卡片允许的最小体面宽度”。

---

## 12. 何时优先 Grid，何时优先 Flex（强化版）

如果你主要在思考：

- “这一行怎么分？” -> Flex
- “这一块切几行几列？” -> Grid
- “整页骨架 + 局部组件排布” -> Grid + Flex 组合

这个判断比背属性更重要。

---

## 12.1 快速决策树（实战版）

当你准备写布局时，先问自己：

1. 主要是“一行/一列”的分配吗？  
   - 是 -> Flex
2. 需要“行列同时控制”吗？  
   - 是 -> Grid
3. 是“整页骨架 + 组件细节”吗？  
   - 是 -> Grid 负责骨架，Flex 负责内部

这样你会明显减少“写完又推翻重来”的情况。

---

## 13. 小实验：同一需求两种实现

需求：做一个 6 卡片区域，支持响应式。

请各做一版：

1. Flex 版（`flex-wrap`）
2. Grid 版（`repeat(auto-fit, minmax(...))`）

然后比较：
- 代码长度
- 可读性
- 断点数量
- 可维护性

你会直观理解“选对模型”的收益。

---

## 14. 常见面试问法（新手友好版）

### Q1：`flex: 1` 一定等分吗？

不一定。  
如果子项有最小宽度限制、内容过长、或 basis/width 组合不同，结果可能不完全等分。

### Q2：Grid 能完全替代 Flex 吗？

不建议这么想。  
二者是分工关系，不是替代关系：
- Grid 擅长二维骨架
- Flex 擅长一维分配

### Q3：为什么我用了 Grid 还是写了很多媒体查询？

可能是你把“结构问题”和“样式微调问题”混在一起了。  
Grid 可以减少一部分断点，但字体、间距、导航交互仍可能需要断点微调。

---

## 15. 5 分钟练习建议

把同一块卡片区各写一版：

1. Flex + `flex-wrap`
2. Grid + `auto-fit/minmax`

然后只做一件事：不断缩放窗口，观察哪一版更少改代码仍能保持可读。  
这比背定义更快建立“算法选型直觉”。
