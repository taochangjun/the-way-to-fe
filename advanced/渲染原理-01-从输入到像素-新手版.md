# 渲染原理 01：从输入到像素（新手版）

> 目标：先建立“全景图”，知道浏览器渲染到底分几步，每一步在解决什么问题。

---

## 1. 为什么要学渲染原理

你已经学过了 `box-model / flex / position / responsive`。  
下一步卡住你的，通常不是“不会写属性”，而是：

- 为什么同样是改样式，有时候页面会卡？
- 为什么 `fixed` 有时“不固定”？
- 为什么 `z-index` 写了也不生效？

这些问题都和浏览器渲染流程有关。

---

## 2. 浏览器渲染六步（先记这个）

浏览器把代码变成屏幕像素，大致经过 6 步：

1. 解析 HTML -> DOM
2. 解析 CSS -> CSSOM
3. 样式计算（Style）-> 每个元素最终样式值
4. 布局（Layout）-> 每个元素的尺寸和坐标
5. 绘制（Paint）-> 把颜色、边框、文字画出来
6. 合成（Composite）-> 图层合并后显示到屏幕

大白话：

- 前 4 步更像“算数学题”
- 后 2 步更像“真的拿笔画”

---

## 2.1 术语翻译表（先不用死记）

第一次看渲染文章最容易被英文吓到，先把词翻成白话：

- `DOM`：页面结构清单（有哪些元素）
- `CSSOM`：样式规则清单（每类元素怎么长）
- `Style`：把“规则清单”算成每个元素的最终样式
- `Layout`：给每个元素算“大小 + 坐标”
- `Paint`：把背景、边框、文字画出来
- `Composite`：把多层画面叠在一起，显示到屏幕
- `Reflow`：重新算布局
- `Repaint`：重新画外观

你现在不需要背定义，先知道“它们各管哪件事”就够了。

---

## 3. 用你熟悉的页面做类比

以一个卡片页为例：

- DOM：页面上有哪些节点（header、card、button）
- CSSOM：这些节点有哪些样式规则
- Style：`width` 最终到底是 `200px` 还是 `50%` 算出来的值
- Layout：每张卡片到底在屏幕的哪个位置
- Paint：卡片背景色、边框、阴影怎么画
- Composite：如果有多个图层，最终怎么叠到一起

这就是“从声明到像素”的完整链路。

---

## 3.1 一个完整小例子：浏览器到底在算什么

假设你写了：

```html
<div class="card">Hello</div>
```

```css
.card {
  width: 50%;
  margin: 0 auto;
  padding: 16px;
  background: #fff;
}
```

浏览器大致在做这些事：

1. 在 DOM 里找到这个 `div`
2. 在 CSSOM 里找到 `.card` 规则
3. 样式计算：确认 `width=50%`、`padding=16px` 等最终样式
4. 布局计算：如果父容器是 800px，那内容宽约是 400px（再结合盒模型求最终占位）
5. 绘制：画白色背景、文字、边界
6. 合成：把结果和其他图层叠好后显示

这就是为什么“看起来一句 CSS”，背后其实是几步连续计算。

---

## 4. 你现阶段最该关注哪两步

新手最有收益的是：

1. **Style（样式计算）**
2. **Layout（布局计算）**

因为你现在写的 CSS 问题，80% 出在这两步：

- 选择器谁生效（优先级）
- 盒模型怎么算（宽高 + padding + border）
- 定位参照系是谁（containing block）
- 元素是否脱离文档流

---

## 5. 重排与重绘（先有概念）

- **重排（Reflow/Layout）**：尺寸或位置变了，要重新算几何信息（开销大）
- **重绘（Repaint）**：外观变了但几何不变（开销较小）

示例：

- 改 `width` -> 常触发重排
- 改 `color` -> 常触发重绘

## 8. 每一步再展开一点（进阶预告）

为了避免“知道名字但不知道做了什么”，这里把 6 步再压实一下：

1. **HTML -> DOM**  
   浏览器把文本解析成节点树。这里会发生容错修复（比如标签没闭合时的自动补全）。

2. **CSS -> CSSOM**  
   浏览器把样式规则解析成可匹配的数据结构。复杂选择器会影响匹配成本。

3. **Style（样式计算）**  
   对每个元素做规则匹配、层叠决策、继承处理、最终值计算。

4. **Layout（布局）**  
   结合盒模型与布局模型（常规流/Flex/Grid/定位），算出尺寸与坐标。

5. **Paint（绘制）**  
   生成绘制指令（背景、文本、阴影、边框等），不是“画一次完事”，而是分步骤画。

6. **Composite（合成）**  
   不同图层按顺序叠加，交给 GPU 显示。某些动画只走合成阶段，性能更好。

---

## 9. 新手最容易混淆的三个点

1. **“页面卡 = JavaScript 慢”**  
   不一定。布局和绘制也会卡，尤其大量 DOM + 频繁改样式。

2. **“改颜色也会很卡？”**  
   改颜色通常不触发布局，但可能触发大面积重绘，仍然会有成本。

3. **“加了 transform 一定快”**  
   方向对，但也要看使用方式和图层数量，过度创建图层也会占内存。

---

## 10. 5 分钟验证实验（建议立刻做）

在一个简单页面里做两组操作：

- 组 A：每 16ms 改一次 `left`
- 组 B：每 16ms 改一次 `transform: translateX(...)`

然后开 DevTools Performance 对比：

- A 组通常 Layout 更频繁
- B 组通常更偏向 Composite

你会非常直观地理解“为什么动画推荐 transform”。

## 11. 新手阅读这类文章的正确姿势

给你一个低痛苦策略：

1. 第一遍只看每节标题和结论，别卡在术语

- 这次页面异常发生在“算样式”还是“算布局”还是“绘制”？
- 我改的这个属性，主要影响哪一步？
- 有没有更便宜的改法（比如 `transform` 替代 `left`）？
