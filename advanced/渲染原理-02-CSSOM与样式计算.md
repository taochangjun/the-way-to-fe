# 渲染原理 02：CSSOM 与样式计算

> 目标：搞懂“浏览器如何决定某个元素最终长什么样”。

---

## 1. CSSOM 是什么

HTML 会被解析成 DOM，CSS 会被解析成 **CSSOM**（CSS Object Model）。

你可以理解为：

- DOM：页面结构树
- CSSOM：样式规则树

样式计算时，浏览器会把两者结合，给每个元素算出最终样式。

大白话类比：

- DOM 像“演员名单”（谁上台）
- CSSOM 像“服装规则册”（每类演员穿什么）
- 样式计算像“发服装”（每个演员最终穿哪套）

---

## 2. 样式计算在做什么

对每个元素，浏览器会做三件事：

1. **选择器匹配**：这条规则命不命中当前元素？
2. **层叠决策**：命中多条时，谁赢？
3. **计算最终值**：把相对值算成可用于布局的值

你可以把它理解为三连问：

1. 这条规则轮得到这个元素吗？
2. 轮到多条时谁说了算？
3. 最后落地的数字到底是多少？

---

## 3. 层叠（Cascade）四个判断维度

当多条规则冲突时，常见判断顺序（简化版）：

1. 来源与重要性（如 `!important`）
2. 选择器优先级（specificity）
3. 书写顺序（后写覆盖先写）
4. 继承规则（某些属性可继承）

你之前学的 `.card.badge-card` 为什么常“更容易赢”，就是因为它更具体。

---

## 3.1 一个“冲突判定”小剧场（超实用）

假设你有这三条规则：

```css
.card { color: blue; }
#title { color: red; }
.card { color: green; }
```

一个元素是：

```html
<h2 id="title" class="card">标题</h2>
```

最终通常是红色，因为：

1. 三条都命中
2. `#title` 的优先级高于 `.card`
3. 所以后面那条绿色 `.card` 也盖不掉它

这个例子能帮你记住：  
**先比优先级，再比先后顺序。**

---

## 4. 计算值为什么重要

你写的是：

```css
div {
  width: 50%;
  padding: 20px;
}
```

布局真正使用的是“算完后的结果”：

- `50%` 要先看父容器宽度
- `padding` 要结合 `box-sizing` 决定总宽度关系

这就是为什么“CSS 看起来很短，结果却依赖上下文”。

再补一句最容易踩坑的点：  
你改的是子元素，但真正影响它的，可能是父元素的尺寸、`box-sizing` 或媒体查询分支。

---

## 5. 一个典型依赖链

如果子元素宽度是 `%`，父元素宽度也是 `%`，那就会形成依赖链，最终追溯到视口（viewport）或根元素。

这也是浏览器必须按一定顺序计算布局的原因。

这也是为什么有些“连环百分比布局”在调试时看起来很绕：  
你改一个父容器，下面多层都在重算。

---

## 6. 结合你当前水平的实战提示

你现在常遇到的样式问题，优先这么查：

1. 规则命中了吗？（Elements 面板）
2. 被谁覆盖了？（查看划线规则）
3. 最终值是多少？（Computed 面板）
4. 这个值依赖谁？（父容器、媒体查询、`box-sizing`）

额外建议：
- 用 DevTools 的 **Computed** 面板看最终值（不要只看你写的值）
- 用 **Styles** 面板看覆盖链路（谁被划线、谁生效）

---

## 7. 这一篇的结论

CSSOM + 样式计算的本质是：  
**浏览器在做一场“规则冲突仲裁 + 数值求解”**。

你写 CSS，不是在“直接画页面”，而是在提供一组约束条件，浏览器帮你解题。

## 13. 新手高频误区清单（读完马上避坑）

1. **误区：后写一定赢**  
   - 纠正：优先级更高的规则可以压住后写规则

2. **误区：百分比就是绝对可靠**  
   - 纠正：百分比依赖父级，父级一变，子级全变

3. **误区：看见样式没生效就加 `!important`**  
   - 纠正：先查命中与优先级，再决定是否需要强制覆盖

4. **误区：只看自己写的文件**  
   - 纠正：浏览器还会有默认样式（UA stylesheet）和组件库样式

---

## 14. 5 分钟动手实验（建议）

目标：亲眼看到“优先级 > 先后顺序”。

1. 写一个元素 `id + class` 同时存在
2. 分别写 `#id` 和 `.class` 的冲突样式
3. 把 `.class` 放到文件最后，观察是否能覆盖 `#id`
4. 打开 DevTools，看最终生效来源

做完这一步，你对 CSSOM/层叠的理解会非常稳。

---

## 8. 选择器匹配为什么“从右往左”

以 `div .item span` 为例，浏览器通常先找 `span`，再看它祖先是否满足 `.item`、`div`。

原因：
- 右侧选择器更能快速缩小候选集合
- 避免从全树盲目向下匹配

对新手的实用结论：
- 不必焦虑“写了一个后代选择器就会崩”
- 但在大型页面里，过度复杂选择器确实会增加匹配成本

---

## 9. 层叠（Cascade）再细一点

同一个属性冲突时，简化决策可记为：

1. 是否 `!important`
2. 选择器优先级
3. 源码后写覆盖先写

常见误区：
- 只看“谁写在后面”
- 忽略了优先级本身

你可以用 DevTools 看被覆盖规则（划线）快速定位。

---

## 10. 继承与非继承属性（高频）

不是所有属性都会继承。

常继承：
- `color`
- `font-family`
- `line-height`

常不继承：
- `width`
- `margin`
- `padding`
- `border`

这解释了一个常见现象：父元素字体变了，子元素也变；但父元素 margin 变了，子元素不会“自动继承”。

---

## 11. “计算值”与“使用值”简化理解

学习规范时你会看到 `specified/computed/used/actual` 这几类值，这里给新手简化：

- **你写下的值**：比如 `width: 50%`
- **计算后的值**：结合父级尺寸、默认值、继承后得出的结果
- **真正布局使用值**：布局阶段最终拿去算坐标和尺寸的值

重点不是记术语，而是理解：  
**CSS 值经常需要上下文才能落地。**

---

## 12. 小练习：用一个属性看懂样式计算

练习代码：

```css
.parent { width: 800px; }
.child {
  width: 50%;
  padding: 20px;
  box-sizing: content-box;
}
```

思考：
1. `.child` 内容宽是多少？
2. `.child` 最终占据宽度是多少？
3. 把 `content-box` 改成 `border-box` 后有何变化？

这 3 个问题答清楚，你对样式计算就进入“可推理”阶段了。
